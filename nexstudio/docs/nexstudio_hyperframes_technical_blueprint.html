NexStudio technical blueprint

A complete implementation guide for the NexMind-orchestrated Studio inside NexMarkets: HyperFrames-based video generation, HTML/CSS still-image generation, hidden specialist orchestration, approvals, payment, rendering, revisions, delivery, security and production operations.

01 · Core decision

Build NexStudio as a NexMind product, not as a HyperFrames skin

HyperFrames should be treated as the video composition, preview, editing and rendering substrate. It must sit behind NexMind and the NexMarkets project system. Users should never need to understand HyperFrames, agent skills, HTML composition rules, render workers, audio providers or queue mechanics.

Public truth

The user asks NexMind to create a finished piece. NexMind gathers intent, shows a structured summary, conducts the live refinement, requests explicit approvals, starts production, reports progress and returns versions. Every visible message, approval request and recovery action comes from NexMind.

Internal truth

NexMind is an orchestration façade over deterministic workflows and specialist services. A conversation director routes work to brief, brand, script, storyboard, asset, motion, voice, sound, composition, quality, rendering and delivery specialists. These specialists do not speak directly to the user.

03 · HyperFrames research audit

What HyperFrames is as implemented on 13 July 2026

HyperFrames is HeyGen’s Apache-2.0 open-source framework for turning HTML, CSS, media and seekable animation into deterministic video. Its current public stack includes a CLI, core parser/runtime, capture engine, producer, player, Studio, SDK, shader transitions, reusable catalog, agent skills and multiple cloud deployment paths.

Authoring model

Plain HTML composition files with data-* timing attributes, CSS and seek-safe animation. No React requirement and no proprietary project format.

Render model

Frame-by-frame seeking in headless Chrome, then FFmpeg encoding and audio mixing. Determinism enables CI, regression testing and batch rendering.

Product integration

@hyperframes/sdk supports headless edits, patches, persistence and sparse template overrides. @hyperframes/player embeds previews. @hyperframes/producer renders programmatically.

Version pin for this workspace: @hyperframes/core and @hyperframes/player v0.7.56. HyperFrames is changing quickly; pin exact package versions and run a controlled upgrade test rather than using an unbounded latest version in production.

What HyperFrames does not provide for NexMarkets

  • It does not provide NexMarkets identity, wallet, token eligibility, USDC payment, escrow or entitlement logic.
  • It does not provide the NexMind conversation, live voice room, confirmed-decision panel or public product language.
  • It does not determine what the user is legally allowed to use; NexMarkets must store source rights and consent.
  • It does not replace product-level project, version, approval, revision, notification or delivery workflows.
  • It does not guarantee a polished creative result merely because the composition is valid; creative direction and quality gates remain NexMarkets responsibilities.
  • Its visual editor should not be exposed wholesale as the default NexStudio UX; that would fracture the NexMind experience and reveal implementation complexity.
  • Its fast-changing public API requires version pinning, contract tests and an adapter boundary.

05 · NexMind orchestration

NexMind is the only conversational authority

The orchestration layer must preserve a single voice and a single source of truth. Specialist outputs are proposed changes to project state. They do not become confirmed decisions until NexMind has either received explicit user approval or applied a pre-approved rule.

Conversation memory

Stores raw transcript, extracted decisions, unresolved questions, user corrections, approved fields and rejected options. The structured brief is authoritative; the transcript is evidence and recovery context.

Tool policy

Every internal tool call carries project ID, session ID, user ID, tenant ID, permissions, source-rights status, cost ceiling and idempotency key. No specialist can publish or charge directly.

Confirmed-context panel

Fields move from “gathered” to “confirmed” only after explicit approval. Any material change to message, format, voice, source or price invalidates downstream approval as required.

Public status mapping

Internal technical states are translated into human product states. Example: COMPOSITION_CHECK_FAILED becomes “I found an issue in the layout and I’m correcting it before production.”

07 · User and production flow

The exact NexStudio journey

A. Studio entry and initial capture

  1. User enters Studio and sees “Create a video or make information visual.”
  2. User chooses Video or Infographic / still image.
  3. The interface asks what finished piece is needed, where it will be used and what should be understood or acted on.
  4. User adds one or more sources: website, X post, announcement, social post, pasted text, files or an existing Marketplace work item.
  5. NexMind extracts a preliminary brief and shows a concise structured summary: outcome, audience, message, format, source, tone, brand, CTA and missing decisions.
  6. The primary next action is clearly visible: Speak with NexMind to refine this. It must not be hidden in a corner or secondary utility.

B. Live refinement

  1. Open the immersive NexMind live room with distinct listening, user-speaking, thinking and NexMind-speaking states.
  2. Show live transcript, current confirmed decisions, source context and the next question.
  3. NexMind asks only high-value questions. It should not repeat fields already answered or ask technical questions users do not need to decide.
  4. At the end, NexMind presents the production direction: message, visual approach, voice, pace, source usage, output format and CTA.
  5. User approves or requests changes. Approval creates an immutable CreativeDirection version.

C. Video purchase and production

  1. Generate one representative still hero frame from the approved direction. This is not a motion preview and must be watermarked or resolution-limited if policy requires.
  2. Calculate server-side price: 5 USDC, or 4 USDC if the connected wallet currently satisfies the 50,000 NEX threshold.
  3. Show exactly what is included: 30-second video, selected aspect ratio, approved voice/mix, final MP4, version record and defined revision entitlement.
  4. Collect payment or escrow authorization.
  5. Require Telegram notification selection/confirmation before the workflow can enter production.
  6. Start durable production. The user may leave. Dashboard and Creation Detail continue to show progress.
  7. When complete, run final quality checks, persist deliverables, send Telegram notification and show review actions.

D. Still-image purchase and production

  1. After brief and design-direction approval, show the 0.10 USDC charge. There is no free final preview.
  2. After payment, generate the HTML/CSS composition and render the final image at the selected approved size.
  3. Run clipping, overflow, contrast, text-fit, logo-clear-space, source and export checks.
  4. Return PNG by default; optionally create WebP/JPEG derivatives without replacing the master PNG.

E. Delivery and reuse

  1. The result opens in Creation Detail with version, sources, production decisions, deliverables and revision history.
  2. User can download, duplicate, revise, add to a Listing, attach to active work, keep in Creations or add approved work to a NexCard.
  3. Public publishing and NexCard attachment require explicit approval for that exact artifact version.

Database rules

  • Use row-level tenant scoping on every project, source, asset, approval, payment and deliverable query.
  • Artifact files are immutable. Revisions create new versions and preserve ancestry.
  • Approval references an exact version ID, never “the current version”.
  • Payment quote stores wallet snapshot, NEX eligibility result, price, expiry and pricing-rule version.
  • Render callbacks are idempotent and may arrive more than once or out of order.
  • Every internal specialist output is stored with prompt/template version, model/provider version, source IDs and trace ID.

Scene composition rules

  • Every visible editable element receives a stable unique id.
  • CSS defines the hero-frame resting layout. Entrances animate from an offset to the CSS position.
  • All animation must be seek-safe and registered with the composition timeline.
  • Do not call video.play(), pause() or set media current time manually; HyperFrames owns media lifecycle.
  • Freeze external assets locally before checking or rendering. Final bundles should require no arbitrary internet access.
  • Keep scene exits visible into the transition; avoid fading every scene to nothing before transition unless the final creative direction explicitly needs it.
  • Use nested compositions to isolate scenes and make revision scope predictable.
  • Keep the 30-second total exact; fail build if scene durations do not sum to 30 seconds.

import { openComposition } from "@hyperframes/sdk";

export async function applyRevision(
  html: string,
  ops: RevisionOperation[],
): Promise<{ html: string; patches: unknown[] }> {
  const composition = await openComposition(html, { history: true });
  const captured: unknown[] = [];
  const unsubscribe = composition.on("patch", ({ patches, origin }) => {
    captured.push({ patches, origin });
  });

  composition.batch(() => {
    for (const op of ops) {
      switch (op.type) {
        case "replace_text":
          composition.setText(op.elementId, op.text);
          break;
        case "set_style":
          composition.setStyle(op.elementId, op.styles);
          break;
        case "set_timing":
          composition.setTiming(op.elementId, op.timing);
          break;
        case "replace_asset":
          composition.setAttribute(op.elementId, "src", op.assetUri);
          break;
        default:
          assertNever(op);
      }
    }
  });

  const updated = composition.serialize();
  unsubscribe();
  composition.dispose();
  return { html: updated, patches: captured };
}

export async function produceVideo(ctx: WorkflowContext): Promise<void> {
  const project = await projects.requireState(ctx.projectId, "READY_FOR_PRODUCTION");
  await gates.requirePaid(project.id);
  await gates.requireTelegram(project.id);
  await locks.acquire(`production:${project.id}`, ctx.idempotencyKey);

  await projects.transition(project.id, "PRODUCING");

  const sources = await activities.ensureSourceBundle(project.id);
  const brand = await activities.resolveBrandKit(project.id);
  const direction = await activities.requireApprovedDirection(project.id);
  const script = await activities.generateAndValidateScript({ sources, brand, direction, seconds: 30 });
  const storyboard = await activities.generateAndValidateStoryboard({ script, direction, seconds: 30 });
  const audio = await activities.produceAudio({ project, script, storyboard });
  const assets = await activities.resolveAndFreezeAssets({ project, sources, storyboard, brand });
  const bundle = await activities.buildHyperframesBundle({ project, brand, direction, storyboard, audio, assets });

  const check = await activities.checkComposition(bundle);
  if (!check.pass) {
    const repaired = await activities.repairComposition({ bundle, findings: check.blockingFindings });
    const recheck = await activities.checkComposition(repaired);
    if (!recheck.pass) throw new WorkflowBlockedError("COMPOSITION_CHECK_FAILED", recheck);
  }

  await projects.transition(project.id, "CHECKING");
  const render = await activities.submitRender({ bundle, idempotencyKey: `render:${bundle.sha256}` });
  const master = await activities.awaitRenderAndIngest(render);
  const finalQa = await activities.runFinalVideoQa({ master, storyboard, audio, brand });
  if (!finalQa.pass) throw new WorkflowBlockedError("FINAL_QA_FAILED", finalQa);

  const deliverables = await activities.createDeliverables({ project, master, finalQa });
  await creations.attach(project.id, deliverables);
  await projects.transition(project.id, "READY");
  await notifications.sendReady(project.id);
}

import { chromium } from "playwright";
import { mkdir, readFile } from "node:fs/promises";
import path from "node:path";

export async function renderStill(input: {
  htmlPath: string;
  outputPath: string;
  width: number;
  height: number;
  deviceScaleFactor?: number;
}): Promise<void> {
  await mkdir(path.dirname(input.outputPath), { recursive: true });
  const browser = await chromium.launch({
    headless: true,
    args: ["--disable-dev-shm-usage", "--font-render-hinting=none"],
  });

  try {
    const context = await browser.newContext({
      viewport: { width: input.width, height: input.height },
      deviceScaleFactor: input.deviceScaleFactor ?? 1,
      colorScheme: "dark",
      reducedMotion: "reduce",
    });
    const page = await context.newPage();
    page.on("console", msg => { if (msg.type() === "error") throw new Error(msg.text()); });
    page.on("pageerror", error => { throw error; });

    await page.goto(`file://${path.resolve(input.htmlPath)}`, { waitUntil: "networkidle" });
    await page.evaluate(async () => { await document.fonts.ready; });
    await page.waitForFunction(() => (window as any).__STILL_READY__ === true, null, { timeout: 15_000 });

    const problems = await page.evaluate(() => {
      const all = [...document.querySelectorAll<HTMLElement>("body *")];
      return all.flatMap(el => {
        const r = el.getBoundingClientRect();
        const issues: string[] = [];
        if (r.left < -0.5 || r.top < -0.5 || r.right > innerWidth + 0.5 || r.bottom > innerHeight + 0.5) {
          issues.push(`${el.id || el.tagName}: outside canvas`);
        }
        if (el.scrollWidth > el.clientWidth + 1 || el.scrollHeight > el.clientHeight + 1) {
          issues.push(`${el.id || el.tagName}: clipped content`);
        }
        return issues;
      });
    });
    if (problems.length) throw new Error(`Still QA failed: ${problems.join("; ")}`);

    await page.screenshot({ path: input.outputPath, type: "png", fullPage: false, animations: "disabled" });
    await context.close();
  } finally {
    await browser.close();
  }
}

HyperFrames intentionally executes HTML and JavaScript in a browser renderer. NexMarkets must add a production sandbox, network policy and source-ingestion controls. A valid composition is not automatically a safe composition.

Render sandbox

Use ephemeral containers or microVMs with no cloud metadata access, read-only base image, non-root user, CPU/memory/time limits, isolated temporary storage and a strict outbound network policy.

Source sandbox

Website capture runs separately from rendering. Block private IP ranges, metadata endpoints, localhost, file schemes, non-HTTP protocols and redirects into restricted ranges.

Asset security

Virus-scan uploads, validate MIME by content, strip active content where appropriate, cap dimensions/duration, normalize media and store immutable hashes.

Tenant isolation

Every signed URL, queue job, render bundle and callback carries tenant/project scope. Workers receive only the minimum project assets required for that job.

Network policy

  • Final render bundles must default to no outbound network access.
  • All fonts, images, video, audio, Lottie and 3D assets are downloaded, validated, hashed and rewritten to local bundle paths.
  • Allowlisted CDN scripts should be vendored into the worker image or project bundle rather than fetched at render time.
  • Website capture uses a separate service with browser egress controls and response-size/time limits.
  • Never pass arbitrary user-provided command-line flags or file paths into shell commands.

Secrets and provider access

  • Store provider credentials in a secret manager and inject only into the worker that needs them.
  • Use separate credentials and billing projects for development, staging and production.
  • Do not store provider keys in project bundles, HTML, client code, transcripts or logs.
  • Log provider operation IDs, not secret-bearing URLs or complete request bodies containing user assets.
  • Rotate webhook secrets and verify timestamp, signature and replay window.

License boundary

HyperFrames itself is Apache 2.0, but every dependency, catalog block, font, stock asset, voice, music track, generated asset and cloud service can have separate terms. Maintain a software bill of materials, preserve required notices, and treat provider usage terms and media rights as separate from the HyperFrames license.

20 · Ordered build runbook

Implementation sequence for the developer

Build in this order. Each phase has a hard exit condition so frontend polish does not hide missing production logic.

Phase 1 — Domain foundation

  1. Create StudioProject, CreativeBrief, SourceRecord, BrandKit, CreativeDirection, Storyboard, Approval, Quote, PaymentIntent, ArtifactVersion, RenderJob, RevisionPlan and AuditEvent schemas.
  2. Implement state transition service with allowed transitions and transactional event publication.
  3. Implement S3-compatible immutable storage keys and signed access.
  4. Implement tenant authorization and audit logging.

Exit: a project can be created, versioned, approved and recovered without any generation provider.

Phase 2 — NexMind session and live refinement

  1. Bind each NexMind session to a Studio project.
  2. Implement structured brief extraction and field confirmation.
  3. Build the summary view and prominent “Speak with NexMind” action.
  4. Build live-room states, transcript, confirmed-decisions panel and disconnect recovery.
  5. Ensure public language never exposes internal models/agents/providers.

Exit: a user can move from outcome to an approved CreativeDirection entirely through NexMind.

Phase 3 — Source and brand ingestion

  1. Implement file upload, text, URL, X post and Marketplace work adapters.
  2. Add SSRF controls, file scanning, normalization and rights attestation.
  3. Build BrandKit resolution from user/project assets and prior approved creations.
  4. Generate structured camera-ready design rules and frozen asset manifests.

Exit: every claim and asset in a test brief traces to a stored source/rights record.

Phase 4 — Still engine first production slice

  1. Create the template registry and variable schema.
  2. Implement payment gate for 0.10 USDC.
  3. Implement Playwright renderer, font readiness and overflow checks.
  4. Create master PNG storage, derivatives and Creation Detail delivery.

Exit: paid still projects render deterministically at all supported dimensions with golden tests.

Phase 5 — HyperFrames adapter and preview

  1. Pin HyperFrames and create the internal adapter package.
  2. Implement bundle builder with root + scene compositions, local assets and manifest.
  3. Implement hyperframes check integration and diagnostic artifact capture.
  4. Embed the HyperFrames player in an authenticated same-origin preview route.
  5. Create the single hero-frame preview path.

  1. Implement Kokoro-first voice service and Gemini exception path.

Phase 8 — Scale and operations

  1. Add provider health, queue autoscaling, budget ceilings and cost attribution.
  2. Add golden upgrade pipeline and previous-version worker images.
  3. Add load tests, failure injection and support diagnostics.
  4. Add optional controlled visual fine-edit mode using HyperFrames Studio/SDK components.
